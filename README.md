# image-compressor

A JPEG-style image compressor I wrote in C++17 from scratch, no external image or math libraries. It takes an uncompressed 24-bit BMP, runs it through a DCT-based compression pipeline, and writes it out in a small custom binary format (`.icc`), with a decoder to get a BMP back out.

I'm a second-year EE undergrad and this started as a way to actually build something with the signals & systems stuff I'd been learning in class, instead of just doing problem sets. The core of JPEG compression is a frequency-domain transform (the DCT is basically a cousin of the Fourier transform), quantization (same idea as ADC quantization noise from electronics), and exploiting the fact that human vision cares a lot less about color resolution than brightness resolution (which is its own kind of sampling-rate tradeoff). Once I saw that connection it felt like a good project to actually implement rather than just something in a textbook.

## How it works

1. **Read the BMP** into an RGB buffer.
2. **Color transform**: RGB -> YCbCr (BT.601 coefficients). Y is luminance (brightness), Cb/Cr carry the color information.
3. **4:2:0 chroma subsampling**: average every 2x2 block of Cb and Cr down to a single sample. This throws away 75% of the raw chroma samples, based on the fact that our eyes are much more sensitive to brightness detail than color detail.
4. **8x8 block DCT**: each plane gets split into 8x8 blocks, and each block goes through a separable 2D DCT-II (two passes of a 1D DCT, using a precomputed cosine basis matrix — no FFT library, just direct matrix multiplication). This concentrates most of a block's energy into a handful of low-frequency coefficients.
5. **Quantization**: divide each DCT coefficient by an entry in a quantization table (the standard JPEG luma/chroma tables, scaled by a quality setting from 1-100). This is where most of the actual compression comes from — a lot of the small high-frequency coefficients round down to zero.
6. **Zigzag scan**: reorder the 64 coefficients in each block from a low-to-high frequency diagonal pattern, so the zeros end up clustered together.
7. **RLE**: the DC coefficient is delta-encoded against the previous block's DC (since neighboring blocks tend to have similar average brightness/color). The 63 AC coefficients are run-length encoded as (zero-run, value) pairs with an end-of-block marker, using a variable-length integer encoding so small values take less space.
8. Everything gets written after a 16-byte header (magic number, width, height, and packed quality/subsampling flags).

Decoding just runs all of that backwards.

There's no Huffman/entropy coding on top of the RLE stream (real JPEG has this) — I kept the scope to the DCT/quantization/subsampling pipeline itself, since that's where the interesting signals & systems ideas are.

## Building

Needs CMake and a C++17 compiler.

```
cmake -B build
cmake --build build
```

This builds two things: `imgcomp` (the compressor/decompressor) and `make_test_image` (spits out a synthetic BMP for testing).

## Usage

```
make_test_image test.bmp 256

imgcomp compress test.bmp test.icc 75
imgcomp decompress test.icc out.bmp
```

Quality is 1-100, defaults to 75. Add `--gray` to `compress` to skip the color pipeline entirely and only keep luminance (useful if you just want to see the raw DCT/quantization effect without chroma subsampling in the mix).

## Results

Measured on a 256x256 synthetic test image (gradient background + solid shapes + a bit of noise, generated by `make_test_image`), original size 196,662 bytes:

| Quality | Compressed size | Ratio | PSNR |
|---|---|---|---|
| 30 | 6,424 B | 30.6:1 | 30.1 dB |
| 50 | 9,142 B | 21.5:1 | 30.8 dB |
| 75 | 17,100 B | 11.5:1 | 31.9 dB |
| 90 | 39,378 B | 5.0:1 | 32.8 dB |

(PSNR here is computed on the full RGB image after 4:2:0 chroma subsampling and reconstruction.)

In `--gray` mode, where color is dropped instead of subsampled, quality 75 gets closer to 14.7:1 with a luma PSNR of 38 dB, since there's no color channel data to store at all.

## What's not here

Huffman coding, progressive/arithmetic coding, anything other than 24-bit BMP as input, a GUI. This is meant to demonstrate the transform-coding pipeline, not be a drop-in JPEG replacement.
